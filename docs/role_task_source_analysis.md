# WhatsApp 角色与任务来源规则

本文件基于用户提供的历史 WhatsApp 导出文本整理，用于后续判断哪些消息可以生成任务、哪些消息只能作为反馈、追问或证据。

## 核心结论

系统不要把所有消息都当成任务来源。任务来源应优先来自明确派工人员的高置信派工消息；其他消息用于更新任务状态、补充证据、生成提醒或记录沟通轨迹。

推荐主线：

```text
高置信派工消息 -> 生成 work_schedules
同事完成汇报 -> 匹配已有任务，更新完成状态
Henry 历史追问/验收消息 -> 作为跟进规则样本，匹配已有任务并沉淀追踪事件
系统自动跟进 -> 扫描任务和汇报，生成缺资料/未回复提醒
附件消息 -> 作为任务证据
临时改派/取消消息 -> 更新已有任务，不重复新建
```

## 角色识别

### Henry atl

职责判断：历史上最关键的跟进、验收、缺资料追问、未回复检查人员。系统设计目标是把这类检查自动化，后续不应依赖 Henry 每天继续发言。

典型发言模式：

```text
Brian, 22-May-2026, 商场26, 未覆返野上 Group
Kelvin, 22-May, 商场4, 有冇维修报告扫描?
有冇 换前 换中 换后 photo record?
咩情况? 有冇跟进结果?
仲未收到你回复 (第2次问)
```

系统处理规则：

- 不要默认把 Henry 的追问消息生成新任务。
- 应优先匹配已有任务，更新为 `未回复`、`资料不足` 或 `需要跟进`。
- 如果匹配不到已有任务，建立“待核对追踪事件”，不要直接写入已确认计划任务。
- 如果 Henry 的消息出现明确派工句式，例如“@同事 明天去某地点检查某问题”，才可以按高置信派工处理。
- 后续正式运行时，Henry 的历史消息主要用于校准规则；每天应由 `/api/followups/run` 自动生成提醒。

### Dicky Company

职责判断：主要即时派工人员，最稳定的高置信任务来源。

典型发言模式：

```text
@Brian Company ono team 商场27 call, 过去看看
@~Karl~華 K11 Atelier call, 7/F 男厕拍卡门锁坏
@kelvin chan Company 抽时间去睇睇
@lin atl 商场41 call, 控制室有一个 CCTV mon 不停闪
```

系统处理规则：

- Dicky 的明确派工消息应直接生成 `work_schedules`。
- 必要条件：有被指派同事、有地点/项目、有任务动作。
- 如果消息包含“改去”“改为”“不用去”“取消”，应优先更新已有任务，而不是新建重复任务。

### Rex Atl

职责判断：协调/派工候选人员，常提供客户现场信息、到场时间、协调要求。

典型发言模式：

```text
@lin atl 明早商场41 10:00 到场协调做检查
商场3 ECALL ... @Karl @Brian 麻烦安排检查
商场3 call Tower A NVR2 又出现故障，需安排师傅到场检查
```

系统处理规则：

- Rex 的明确派工消息可以生成任务，但优先级低于 Dicky。
- 如果没有明确被指派同事，只能生成“待分配任务候选”，不能当作已分派任务。

### Ono atl

职责判断：协调/调度/升级处理人员。

典型发言模式：

```text
@Dicky Company urgent, pls help to arrange
@kelvin chan Company 可以先过去?
如果林海未去，就明天先安排再去
```

系统处理规则：

- Ono 的消息多用于协调和改派。
- 有明确同事、地点、动作时可以生成或更新任务。
- 出现“明天先安排”“改为”等语义时，优先更新任务日期或状态。

### +852 6909 6998

职责判断：疑似派工/接 call/现场协调人员，但身份未确认。

典型发言模式：

```text
商场33 call ...
The hari call ...
@Brian 麻烦你睇睇
而家过去
```

系统处理规则：

- 目前不要默认纳入派工管理人员名单。
- 需要用户确认该号码对应姓名和角色后，再加入 `DISPATCH_MANAGER_SENDERS`。
- 在未确认前，其消息可以作为“任务候选”或“现场线索”，不应自动生成已确认任务。

### 普通同事/维修人员

包括但不限于 Brian、Kit、Kelvin、Lin、Kei、Sam、Kenny、Harris、Foon、Johnny 等。

职责判断：主要提供完成汇报、现场情况、附件、维修结果。

典型发言模式：

```text
例检完成
测试正常
更换后恢复正常
维修报告后补
离场
<媒体已略去>
```

系统处理规则：

- 默认不从普通同事的完成汇报反推新任务。
- 应用这些消息匹配已有任务，并更新完成状态、缺失资料、附件证据。
- 如果普通同事偶尔派工给别人，先作为低置信候选，除非后续用户确认其具备派工权限。

## 任务来源优先级

### 高置信任务来源

同时满足以下条件，才自动生成 `work_schedules`：

```text
1. 发言人是已确认派工人员
2. 消息中有明确 @ 被指派同事
3. 消息中有地点或项目
4. 消息中有任务动作，例如 call、过去看看、安排检查、明早到场、urgent
5. 消息不是取消、追问、完成汇报
```

### 中置信任务来源

```text
1. 发言人是协调人员，但没有明确 @ 同事
2. Henry 的追问消息里引用了过往任务
3. 电话号码发出的 call 消息，但身份未确认
```

处理方式：进入候选或追踪事件，不直接生成已确认计划任务。

### 低置信任务来源

```text
1. 普通同事完成汇报
2. 媒体占位
3. 单纯说“正常”“完成”“离场”
4. 没有地点或没有执行对象的聊天
```

处理方式：只作为反馈和证据，不生成任务。

## 后续实现建议

建议数据库后续增加任务事件表，例如：

```text
task_events
- task_id
- raw_message_id
- event_type: dispatch / followup / reply / attachment / reminder / cancel / reschedule
- sender
- event_text
- created_at
```

这样可以把 Henry 的追问、同事回复、附件、提醒记录都挂到同一个任务下面，避免污染 `work_schedules`。

当前系统已经实现 `task_events` 基础表，用于保存 Henry 的未回复、缺 PDF、缺照片、问跟进结果等历史追踪事件；同时已经提供 `/api/followups/run` 自动跟进接口，用于不依赖 Henry 发言地生成未回复和资料不足提醒。后续还需要把普通同事回复、附件、提醒发送结果也逐步接入该表。

## 客户可配置角色

人员职责不应写死在代码中。客户可以在本地管理页面维护人员角色、别名和处理原则：

```text
http://127.0.0.1:8000/admin/settings
```

当前角色：

```text
dispatch_manager: 派工人员
followup_manager: 跟进/验收人员
technician: 维修执行人员
issue_reporter: 问题上报人员
viewer: 管理查看人员
```

处理原则：

```text
派工人员的明确派工消息 -> 可自动生成正式任务
普通成员发现的问题 -> 先记录为待确认线索，不直接变成正式任务
派工人员后续确认 -> 转成正式任务
派工人员说不用处理 -> 标记忽略/关闭
长时间无人确认 -> 提醒管理人员确认
```
