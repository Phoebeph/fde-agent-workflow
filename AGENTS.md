# AGENTS.md

## 目标与优先级

本文件用于约束代理在本项目中的执行边界、交付质量与可追溯性，目标是让任务可直接执行、结果可验证、过程可回溯。

优先级规则必须按以下顺序执行：

1. 用户当前明确要求
2. 安全规则
3. 本文件规则
4. `README.md` 与其他项目文档

如果用户要求与安全规则冲突，必须优先遵守安全规则。

## 执行原则

- 必须先理解任务目标、约束与预期结果，再开始修改。
- 必须遵循现有项目结构、命名风格与接口约定。
- 必须保持最小改动，只修改与当前任务直接相关的内容。
- 必须优先复用现有函数、组件、模块与项目模式。
- 必须对外部输入做校验，并明确处理异常路径。
- 必须保证输出可运行、可维护，不得伪造结果或编造上下文。
- 必须保护敏感信息，不得写入代码、日志、注释或文档。
- 禁止无理由新增依赖、删除文件、修改 Git 历史。
- 建议在执行命令前先检查 `README.md`、`package.json` 与项目文档中的既有命令。
- 建议仅在必要时补充日志与注释；新增注释必须使用英文。

## 开发流程

1. 理解任务：确认目标、输入输出、约束和验收标准。
2. 消除歧义：存在关键不确定项时，先提出简洁澄清问题。
3. 制定方案：优先选择最简单、最直接、最可靠的最小可行修改。
4. 实施改动：按现有模式实现，避免超范围重构。
5. 自检验证：执行必要验证并记录结果；若失败，必须说明原因和下一步建议。
6. 交付说明：说明做了什么、改了什么、如何验证、是否存在风险。

## 回复规范

任务不清晰时，必须先提澄清问题，不得直接假设并实现。

任务清晰时，最终回复必须包含以下最小字段：

1. 本次做了什么
2. 修改了哪些文件
3. 验证结果
4. `summary.md` 更新状态
5. 后续建议（如有）

修复类任务，必须说明：问题原因、修改逻辑、影响范围。

新增功能类任务，必须说明：新增能力、核心实现、验证方式。

## summary.md 规范

默认规则：每次完成用户请求并最终回复前，必须更新项目根目录 `summary.md`。

唯一例外：若用户明确要求本次不要更新 `summary.md`，可跳过更新，但最终回复必须明确说明未更新原因。

`summary.md` 不存在时，必须先创建并写入以下初始内容：

```markdown
# Project Summary

本文件用于记录每次用户请求和 AI 回复的简要总结，方便后续恢复项目上下文。
```

`summary.md` 已存在时：

- 必须只在文件末尾追加。
- 禁止覆盖、删除、重排既有记录。

每次追加记录必须使用以下模板：

```markdown
---

## YYYY-MM-DD HH:mm

### 用户请求

- 简要总结用户这次提出的问题或任务。

### AI 回复

- 简要总结本次回答、分析、修改、实现方案或结论。

### 修改文件

- 如果有修改文件，列出文件路径和修改内容。
- 如果没有修改文件，写：无文件修改。

### 重要说明

- 记录重要决策、命令、限制、后续待办或注意事项。
```

记录要求：

- 必须每次请求追加一条记录。
- 即使无代码改动，也必须记录结论。
- 必须避免写入敏感信息；遇到敏感内容时必须抽象描述。
- 如果更新失败，最终回复必须明确说明失败原因。

## 禁止事项

- 禁止重写无关代码或做超范围重构。
- 禁止未经要求修改公开接口或行为语义。
- 禁止引入不必要抽象层与复杂度。
- 禁止引入 mock/假数据/占位实现（除非用户明确要求）。
- 禁止在未执行验证的情况下声称“已验证”或“已测试”。
- 禁止将未知项目细节当作既定事实。
